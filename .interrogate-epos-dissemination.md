# Destination: EPOS RINEX Dissemination from receivers (phase 1)

## Problem
The legacy `epos-gnss` swarm container can't produce RINEX3 long names and needs
manual intervention whenever TOS changes (corrections, new instruments, `in_epos`
flag flips). We retire it by running EPOS dissemination entirely from `receivers`,
delivering the capability the old library lacks â€” R3 **long-name** files on the EPOS
files server â€” while keeping EPOS fed throughout. Phase 1 keeps the existing EPOS
gnss-europe DB (md5 index); a later phase 2 migrates the index to sha256 + gps_health.

## In scope (phase 1)
- `receivers epos-disseminate` (CLI + scheduled, **double-gated `active:false`**),
  extending the `sync.yaml` / `SyncTarget` infra with a `tier:dissemination` target:
  TOS include-filter (`in_network_epos=true`), rename/convert spec, EPOS files-server dest.
- **Convert+rename stage** before rsync: source precedence = prefer archived RINEX,
  else rinex-from-archived-raw; convert R2â†’R3 + **long RINEX3 name**
  (`gtimes.rinex3_filename` + gfzrnx/CRX2RNX/RNX2CRX from gps-tools), **header set from
  TOS**. Output **cached keyed on `hash(source content_sha256 + TOS-metadata
  fingerprint for that epoch)`** â€” NOT source hash alone, so a header correction
  invalidates exactly the affected files and forces a re-render (convert once per
  (content, metadata) pair).
- **Retroactive header-correction re-push:** a TOS correction that changes RINEX
  header content must **re-push affected historical files back in time** â€” recompute
  the affected date range from the corrected attribute's `date_from`/`date_to`,
  re-convert (cache auto-invalidated via the metadata fingerprint), **overwrite** the
  R3 files on the EPOS server (rinex is derived/regenerable, so overwrite is allowed
  here â€” unlike the immutable archive), and update the `rinex_file` index
  (`revision_date`). Pushed via the reactive trigger, not a manual replay.
- **tostools header-QC gate** (`gps_metadata_qc`): verify R3 header vs TOS/site log;
  on mismatch flag+skip (never push wrong-header file).
- **Both legacy halves ported:** (a) TOSâ†’EPOS gnss-europe metadata ETL
  (station/coords/monument/bedrock/geological/contact/device-history) via `TOSClient`
  (dead `/tos/v1/`â†’`/tos/internal`), parameterized SQL, surgical per-station upsert (no
  TRUNCATE-all); (b) `rinex_file` md5/md5uncompressed indexer with `gtimes` filename parsing.
- **Reactive TOS sync:** daily TOS-fingerprint-diff poll (+ manual `--refresh-metadata`)
  re-ETLs only changed stations. `in_epos` on â†’ **full-station archive backfill**
  (floor = install date) + DB populate + site log; off â†’ **stop-only** (mark DB rows
  inactive, no new push). A **header-affecting** change additionally triggers the
  retroactive re-push above for the corrected date range (not just a metadata re-ETL).
- **Site logs:** new `gps-sitelogs` git repo (gps-config-data pattern), per-station IGS
  site logs via `tos sitelog`, regenerated/committed on the same TOS-change trigger.

## Out of scope (phase 1)
- gps_health + sha256 migration of the EPOS index (â†’ phase 2).
- Bulk re-rinexing the historical R2 archive in place + its archive-DB rows (backburner;
  R2â†’R3 stays a dissemination-time transient â€” the archive is never rewritten).
- NetRS R2 codeless-C2D special case (flagged, deferred).
- Any purge/delete on the EPOS server or DB (no delete path exists in phase 1).
- General multi-consumer dissemination framework beyond EPOS.
- #34 precedence-engine integration (phase 1 uses simple prefer-rinex-else-raw precedence).

## Definition of done
- Scheduled+CLI `epos-disseminate` pushes EPOS-flagged stations as R3 long-name files,
  converting R2â†’R3 (cached) where needed.
- Every pushed file passes the tostools header-QC gate before upload.
- EPOS gnss-europe DB (metadata + device history + `rinex_file` md5) populated via
  `TOSClient`; daily fingerprint-diff re-ETLs only changed stations; `in_epos` onâ†’full
  backfill, offâ†’stop-only.
- Per-station site logs generated to `gps-sitelogs` on the trigger.
- A header-affecting TOS correction re-converts + overwrites the affected historical R3
  files on EPOS and updates their `rinex_file` rows (`revision_date`); the convert cache
  invalidates correctly via the metadata-fingerprint key.
- Old swarm container retired after sample-station parity is verified against it.
- All gated `active:false` â€” deploying the code changes nothing until flipped.

## Hard constraints
- Runs on rek-d01 (fast path to ananas RO mount); rawdata stays sole archive writer.
- Reuse, don't reinvent: `archive.ArchiveSync` transport, `sync.yaml`/`SyncTarget`,
  `gtimes` naming, `tostools.TOSClient`/`gps_metadata_qc`/`tos sitelog`, gps-tools binaries.
- The library can't emit long R3 names â†’ naming must come from receivers (`rinex3_filename`).
